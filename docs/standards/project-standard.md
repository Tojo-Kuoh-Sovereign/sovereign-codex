# Project Standards

The Project Standards establish the universal expectations for all documentation produced within Sovereign Codex.

Every document, regardless of subject, should meet these standards to ensure the knowledge base remains accurate, consistent, maintainable, and trustworthy over time.

These standards define *what* every document should achieve. More specific guidance for writing, structure, evidence, naming, and versioning is provided throughout the remainder of the Standards section.

---

## Core Principles

All documentation published within Sovereign Codex should strive to be:

- Accurate
- Clear
- Consistent
- Maintainable
- Evidence-based
- Reader-focused

These principles apply to every document within the project.

---

## Accuracy

Documentation should represent the best available information at the time it is published.

Where uncertainty exists, it should be clearly identified rather than presented as established fact.

---

## Clarity

Information should be communicated using clear, concise language.

Readers should be able to understand both the purpose of the document and the information it contains without unnecessary complexity.

---

## Consistency

Documentation should follow the project's established standards for writing, formatting, structure, terminology, and presentation.

Consistency improves readability and makes the knowledge base easier to navigate and maintain.

---

## Maintainability

Documentation should be written with future maintenance in mind.

Clear organisation, reusable structures, and well-defined standards reduce the effort required to keep documentation current as projects evolve.

---

## Evidence

Information should be supported by appropriate evidence whenever practical.

Claims, recommendations, and technical guidance should be based upon verified information, with confidence levels communicated where appropriate.

---

## Reader-First Design

Documentation exists to help people solve problems.

Content should prioritise usability, readability, and practical value over unnecessary detail or complexity.

When a choice exists between being technically impressive and being easily understood, clarity should take priority.

---

## Continuous Improvement

Documentation is never considered complete.

As new information becomes available, standards evolve, or improvements are identified, documentation should be reviewed and refined to ensure it continues to represent the highest practical standard.

Project Standards should evolve deliberately through experience, review, and continuous improvement rather than frequent or unnecessary change.
